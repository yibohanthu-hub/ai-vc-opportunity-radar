# 国内实时公开信号管道

## 目的与边界

这条管道不自动做投资决策。它让以国内为优先范围的机会池持续获得新的、可回溯的公开信息，并将信息分为两层：

1. **实时待核验队列**：新闻、产品发布与活动目录中的候选信号；
2. **正式项目 / 活动信息池**：经人工打开来源、补充关键字段并判断后，才可进入。

脚本不会根据标题或摘要编造公司主体、融资金额、估值、创始人、嘉宾或联系方式。找不到就显示“未知 / 待核验”。

## 管道步骤

```text
InfoQ 中文 / 量子位 RSS / 36 氪与雷峰网公开 RSS、融资快报 / 活动行 28 城目录
            ↓
      抓取并记录来源状态
            ↓
国内优先关键词过滤、市场 / 活动新闻排除、标题 / 摘要阶段线索标注
            ↓
     标题 + URL 去重、近 45 天滚动保留
            ↓
  写入 live-signals.js（待核验队列）
            ↓
    浏览器展示来源、时间、入队理由
            ↓
  人工打开来源 → 收藏 / 忽略 / 标签 / 跟进
```

脚本入口：

```bash
npm run refresh:signals
npm run check:signals
```

`refresh:signals` 只在至少取得一条真实候选时才覆盖 `live-signals.js`；若所有来源失败，命令失败并保留上一次成功的队列。单个来源失败时，其他来源仍可更新，页面会显示“部分来源暂不可用”。

## 定时更新与公开部署

公开仓库已启用并完成端到端验证的两个工作流：

- `.github/workflows/refresh-live-signals.yml`：每 4 小时的第 17 分钟运行（北京时间为 00:17、04:17、08:17、12:17、16:17、20:17）；每次成功运行会记录来源状态、最近一次看到时间和出现次数，并提交刷新后的 `live-signals.js`。
- `.github/workflows/deploy-pages.yml`：每次主分支有新提交时验证、构建并部署静态站点到 GitHub Pages。

GitHub Actions 的 `schedule` 是尽力按时，不应被当作金融级 SLA；调度高峰时可能延迟。页面显示最近一次成功抓取时间和每个来源的状态，而不是伪装成实时秒级数据。

## 风险与后续治理

- 公开来源会变更结构、速率限制或停止提供数据；每次运行会暴露来源状态，不能静默当作成功。
- 当前关键词规则只是候选筛选，不能取代实体消歧、地域归属判断、新闻真实性判断或投研核验。
- 下一阶段应增加来源适配器测试、地区 / 主体人工审核、字段级证据、允许名单 / 排除名单，以及来源条款与 robots 要求的定期复查。
