# 作品集案例｜AI Opportunity Radar

> 建议导出为 A4、四页 PDF。以下 `---` 为页间分隔；本案例聚焦产品判断与 0→1 落地，而非把 Demo 写成生产级系统。

---

## 第 1 页｜把投资人“信息过载”改造成一个可体验的产品问题

### 项目背景

国内 VC 投资经理每天要在新闻、创始人网络、活动邀约、数据库、社群和搜索结果之间切换。问题不只是“找不到项目”，而是难以持续判断：哪些公开信号值得点开、信息是否可信、来源在哪里、下一步验证什么，以及如何把今天的发现变成后续跟进。

活动同样需要筛选。一个 AI 峰会可能有高价值创始人交流，也可能只是品牌露出或低信息密度的颁奖环节。投资人需要的不是泛活动列表，而是与“发现项目 / 专家 / 创始人”目标有关的解释。

### 产品目标

我把问题定义为：**帮助 VC 投资经理持续发现国内未上市 AI 项目和高价值 AI 活动，并将公开信息转化为可筛选、可验证、可跟进的机会清单。**

第一版采取“国内优先、海外补充”的覆盖策略：正式样本和实时队列首先服务中国企业的本地业务语境，而不是做一个无差别的全球资讯聚合页。

这个 Demo 必须证明四件事：

1. 我理解投资经理真实的早晚扫描场景；
2. 我能把模糊的投研工作转译为页面、字段和交互流程；
3. 我理解 AI Agent 在信息收集、结构化、初筛、解释和反馈中的合理边界；
4. 我能把方案做成可运行的产品，而不是一份研究 PPT。

### 我负责什么

独立完成产品定义、数据字段设计、公开样本整理、真实公开源接入、规则排序、交互设计、前端实现、浏览器验证与作品集文档。技术上选择零依赖的原生前端加轻量采集脚本，而非先搭建登录、数据库和复杂后端——因为本项目的核心验收是**可操作的产品闭环和可信边界**。

## 第 2 页｜关键取舍：信息池优先，而不是“AI 替你投资”

### 用户流程

```text
Dashboard
  → 项目列表（摘要 + 筛选 + 原因）
  → 项目详情（来源 + 不确定性 + 验证问题）
  → 收藏 / 标签 / 跟进
  → 活动列表（质量判断 + 低价值识别）
  → 收藏与跟进页（批量整理）
```

### 三个产品取舍

| 取舍 | 选择 | 原因 |
| --- | --- | --- |
| “推荐项目”还是“信息池” | 先做信息池 | 投资判断高风险、低可验证；V1 应先覆盖发现、整理、解释与人工验证。 |
| 直接入库还是待核验队列 | 国内公开源先进入候选池，41 个项目（含 18 个早期信号项目）+ 21 个活动作为已整理样本 | 既证明数据采集能力，又不把新闻标题伪装成完整投研档案。 |
| 补全空白还是保留未知 | 公开来源不足时显示“未知 / 待核验” | 这是投研产品的底线；融资额、估值、创始人背景和联系方式不能由模型猜测。 |

### 交互设计

- 项目卡片先展示足以决定“点不点开”的摘要：赛道、创始人、地点、融资线索、B 轮以前可能性、来源更新时间、可信度和一句关注理由。
- 详情页把“为什么值得看”和“为什么不能现在下结论”放在同一视线内：来源、筛选理由、不确定性、验证问题和公开联系入口并列。
- 活动页的低价值标签不是贬低活动，而是相对于当下目标的优先级判断：没有公开创业展示或深度交流机制的品牌活动不应挤占投资经理时间。

## 第 3 页｜Agent 工作流、数据字段与可解释排序

### Agent 工作流

```text
InfoQ 中文 / 量子位 RSS / 36 氪融资快报 / 活动行国内城市目录
  → 候选提取（项目 / 产品 / 活动）
  → 字段规范化
  → 去重（同名 + 官网 + 来源交叉）
  → 可信度与缺失字段标记
  → 规则初筛与理由生成
  → 信息池
  → 投资经理查看、收藏、忽略、标签、跟进
```

### 核心字段

项目实体包含：名称、行业/细分、产品、创始人和背景、所在地、融资阶段/金额/估值/最近融资、B 轮以前可能性、来源、发现与更新时间、可信度、商业化线索、竞争替代、公开联系入口、筛选理由、不确定性和验证问题。

活动实体包含：名称、时间、地点、形式、主办方、主题、嘉宾与质量状态、参与人群、投资人价值、项目发现可能性、来源、更新时间、活动质量和理由。

### 规则，而不是黑箱

项目排序显示六个 1–5 分维度：信息新颖性、来源可信度、信息完整度、AI 相关性、早期阶段线索和潜在跟进价值。

活动排序显示七个维度：主题质量、嘉宾质量、交流形式、投资人实际价值、项目发现可能性、信息可信度和时间紧迫性。

界面明确写出：分数只是公开信号的阅读优先级，**不是投资建议，也不输出“应投 / 不应投”。**

## 第 4 页｜验证、失败案例与下一步

### 验证结果

- 数据检查：41 个国内优先项目、21 个国内优先活动、3 个重复候选合并案例、6 个缺字段案例、25 个需人工核验项目、3 个低价值活动；其中 18 个带有成立时间较短或早期融资信号；另有实时公开源队列的来源状态与字段校验。
- 浏览器验证：Dashboard、项目筛选、项目详情、公开来源、收藏、标签、跟进、活动低价值筛选、活动详情和批量整理全部实际点击通过。
- 响应式检查：桌面和 390px 宽度下均已验证主要首页布局。

### 一个重要失败案例

早期数据整理时，活动页很容易“为了填满字段”而把 2026 年部分会议的地点、嘉宾和日程写成看起来完整但无法确认的事实。这会直接破坏产品可信度。

我改成了三层处理：

1. 只在有官方当前页或已收录公开证据时写具体字段；
2. 不能确认时显示“待官方当前页 / 归档核验”；
3. 将历史会议标记为“历史样本”，不伪装成未来日程。

这比一个“信息更多”但混入猜测的产品更符合投资人工作流。

### 后续迭代

1. 将已上线的国内公开源采集扩展到高校创业营、项目展示、投资机构组合页，并补充抓取版本、字段级证据与人工审核队列；
2. 基于收藏、忽略、标签和跟进做可解释推荐，并允许用户纠正偏好；
3. 接入 CRM / 日历但坚持“人批准后才执行”，不自动触达任何创始人。

### 求职表达

这不是一个金融分析展示，而是一个真实业务工作流的 0→1 产品：我把投资人发现机会的隐性过程拆成数据结构、信息状态、交互决策和可验证边界，并完成一个可实际操作的前端 Demo。
